# Vazon V3 MQTT Topic Registry

Document status: draft
Code status: architecture reference
Scope: Vazon V3 MQTT namespace and payload boundary

## 1. Purpose

```text
This document defines the MQTT topic registry for Vazon V3.
```

It defines topic names, direction, retained policy, payload group, and command result topics.

It does not own module behavior, GPIO logic, UI wording, persistence, or safety rules.

## 2. Ownership References

```text
data model -> docs/v3_data_model.md
module behavior -> docs/v3_contracts/*.contract.md
software boundary -> docs/software_architecture.md
UI presentation -> docs/ui_architecture.md
```

## 3. Base Namespace

```text
vazon/v3/{device_id}/...
```

`device_id` is the stable logical device identifier used by firmware, MQTT broker, and UI.

Exact device_id derivation is not defined in this file.

## 4. Topic Classes

```text
state      device publishes current state
status     device publishes compact status/status_reason
settings   device publishes current settings
cmd        UI/backend publishes command request
ack        device publishes accepted command result
reject     device publishes rejected command result
fail       device publishes failed command result
telemetry  device publishes non-critical service telemetry
service    service/debug channel
```

## 5. Retain Policy

```text
state      retained
status     retained
settings   retained
cmd        not retained
ack        not retained
reject     not retained
fail       not retained
telemetry  not retained
service    not retained
```

Rationale:

```text
UI must receive latest known state after reconnect.
Commands and command results must not be replayed as retained MQTT messages.
```

## 10. Command Payload Envelope

All command topics use one envelope:

```json
{
  "cmd_id": "string",
  "cmd": "string",
  "args": {},
  "source": "ui",
  "ts": 0
}
```

Fields:

```text
cmd_id  unique command id generated by sender
cmd     command name owned by module contract
args    command arguments
source  ui / service / test / unknown
ts      sender timestamp if available
```

## 11. Command Result Topics

```text
vazon/v3/{device_id}/ack/{cmd_id}
vazon/v3/{device_id}/reject/{cmd_id}
vazon/v3/{device_id}/fail/{cmd_id}
```

Result envelope:

```json
{
  "cmd_id": "string",
  "target": "light",
  "cmd": "set_mode",
  "result": "accepted",
  "reason": null,
  "ts": 0
}
```

Result meanings:

```text
ack
    command accepted by owning module or global policy

reject
    command rejected before execution because it is invalid, unauthorized, malformed, or not allowed in current state

fail
    command accepted but failed during execution or output application
```

## 13. Telemetry Topics

```text
vazon/v3/{device_id}/telemetry/connection
vazon/v3/{device_id}/telemetry/firmware
vazon/v3/{device_id}/telemetry/uptime
```

Telemetry is informational and must not be required for local control logic.

## 14. Service Topics

```text
vazon/v3/{device_id}/service/ota/status
vazon/v3/{device_id}/service/provisioning/status
vazon/v3/{device_id}/service/debug
```

Service topics are not normal module state topics.

## 15. Last Will / Availability

Device availability topic:

```text
vazon/v3/{device_id}/availability
```

Payload values:

```text
online
offline
```

Retain policy:

```text
availability retained
```

Broker LWT should publish:

```text
offline
```

Device should publish after successful MQTT connection:

```text
online
```

## 16. Timestamp Rule

```text
ts is a device/runtime timestamp field.
Exact time source is not defined here.
If valid wall-clock time is unavailable, firmware may use monotonic uptime-based timestamp until time is synchronized.
```

## 17. Forbidden

```text
MQTT topic registry does not own module behavior.
MQTT topic registry does not own UI text.
MQTT topic registry does not own GPIO mapping.
MQTT topic registry does not define persistence layout.
MQTT topic registry does not bypass Command Router.
MQTT cmd topics must not be retained.
Command result topics must not be retained.
```
