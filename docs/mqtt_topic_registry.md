# Vazon V3 MQTT Topic Registry

Document status: active minimal
Code status: secure transport, typed codec, and bounded command queue implemented; runtime wiring pending
Scope: MQTT namespace, topic classes, retain policy, and command envelope only

## 1. Purpose

This document defines the minimal MQTT transport surface for Vazon V3.

It does not own module behavior, module field lists, command semantics, GPIO logic, UI wording, persistence, or safety rules.

## 2. Ownership References

```text
module behavior and command names -> docs/v3_contracts/*.contract.md
data-shape rules -> docs/v3_data_model.md
software boundary -> docs/software_architecture.md
UI presentation -> docs/ui_architecture.md
```

## 3. Base Namespace

```text
vazon/v3/{device_id}/...
```

`device_id` is the stable logical device identifier used by firmware, MQTT broker, and UI.

Firmware derives it from the station MAC using the proven project format:

```text
Vazon_{last three STA MAC bytes in lowercase hexadecimal}
```

Example: `Vazon_a1b2c3`.

## 4. Topic Classes

```text
state      device publishes current owner-defined entity state
status     device publishes compact status/status_reason
settings   device publishes current owner-defined settings
cmd        UI/backend publishes command request
ack        device publishes accepted command result
reject     device publishes rejected command result
fail       device publishes failed command result
telemetry  device publishes non-critical service telemetry
service    service/debug channel
```

## 5. Retain Policy

```text
state        retained
status       retained
settings     retained
availability retained
cmd          not retained
ack          not retained
reject       not retained
fail         not retained
telemetry    not retained
service      not retained
```

Commands and command results must never be retained.

## 6. Entity Topic Pattern

Entity topics use canonical entity names from `docs/v3_data_model.md`.

```text
vazon/v3/{device_id}/state/{entity}
vazon/v3/{device_id}/status/{entity}
vazon/v3/{device_id}/settings/{entity}
vazon/v3/{device_id}/cmd/{entity}
```

Examples:

```text
vazon/v3/{device_id}/state/system
vazon/v3/{device_id}/state/climate
vazon/v3/{device_id}/state/pot/0
vazon/v3/{device_id}/state/pot/1
vazon/v3/{device_id}/state/humidifier
```

This file defines the topic pattern, not the payload field list.

## 7. Command Payload Envelope

All command topics use one envelope:

```json
{
  "cmd_id": "string",
  "cmd": "string",
  "args": {},
  "source": "ui",
  "ts": 0
}
```

Fields:

```text
cmd_id  unique command id generated by sender
cmd     command name owned by the target contract
args    command arguments owned by the target contract
source  ui / service / test / unknown
ts      sender timestamp if available
```

The command target is defined by the topic entity.

## 8. Command Result Topics

```text
vazon/v3/{device_id}/ack/{cmd_id}
vazon/v3/{device_id}/reject/{cmd_id}
vazon/v3/{device_id}/fail/{cmd_id}
```

Result envelope:

```json
{
  "cmd_id": "string",
  "target": "fan",
  "cmd": "set_mode",
  "result": "accepted",
  "reason": null,
  "ts": 0
}
```

Result meanings:

```text
ack
    command accepted by owning module or global policy

reject
    command rejected before execution because it is invalid, malformed, unauthorized, or not allowed in current state

fail
    command accepted but failed during execution or output application
```

## 9. Availability

```text
vazon/v3/{device_id}/availability
```

Payload values:

```text
online
offline
```

Retain policy:

```text
availability retained
```

Broker LWT should publish `offline`.

Device should publish `online` after successful MQTT connection.

The transport subscribes once to:

```text
vazon/v3/{device_id}/cmd/#
```

The topic is subscribed with QoS 1. Availability is published retained with
QoS 1, and the broker LWT publishes retained `offline`.

The MQTT connection uses `mqtts` with the ESP certificate bundle. Host, port,
username, and password are supplied by configuration. The transport uses the
firmware-derived `device_id` as its MQTT client ID.

The current transport accepts a complete command payload up to 1024 bytes and
reassembles fragmented ESP-MQTT data events before calling the command
boundary. Oversized or inconsistent fragments are rejected locally.

The command codec validates the canonical topic and envelope, translates
owner-defined serialized arguments into the existing typed module arguments,
routes exactly once through Command Router, and creates the corresponding
non-retained `ack`, `reject`, or `fail` response. Unknown fields and invalid
field types are rejected rather than ignored.

The asynchronous MQTT event callback does not invoke Command Router directly.
It copies each complete command into a bounded runtime queue. The queue
consumer acquires the runtime lock, decodes and routes exactly one command,
releases the lock, and only then publishes the non-retained result. Queue
overflow rejects the incoming work locally and is counted/logged; it never
blocks the MQTT event task.

## 10. Service Topics

Service/debug topics are allowed only for service-owned features:

```text
vazon/v3/{device_id}/service/ota/status
vazon/v3/{device_id}/service/provisioning/status
vazon/v3/{device_id}/service/debug
```

Service topics are not normal module state topics and must not bypass Command Router.

## 11. Timestamp Rule

```text
ts is a device/runtime timestamp field.
Exact time source is not defined here.
If valid wall-clock time is unavailable, firmware may use monotonic uptime-based timestamp until time is synchronized.
```

## 12. Forbidden

```text
Do not list module field payloads here.
Do not list module command names here.
Do not define module behavior here.
Do not define UI text here.
Do not define GPIO mapping here.
Do not define persistence layout here.
Do not retain cmd/ack/reject/fail topics.
```
